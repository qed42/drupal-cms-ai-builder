# REQ: Content Depth Enforcement & Quality Review Agent

**Status:** Draft
**Priority:** High
**Author:** Product Owner
**Date:** 2026-03-20

---

## Problem Statement

Pages generated by the AI content pipeline consistently produce only 2-3 components per page, despite design rules specifying 4-7 sections for content-heavy pages like Home, Services, and About. This results in thin, unconvincing websites that fail to meet the quality bar a real business owner would expect.

Additionally, there is no quality gate between content generation and blueprint finalization. The system generates content in a single pass and commits it — there is no review loop to catch thin content, missing SEO signals, or poor content structure before the blueprint reaches provisioning.

### Root Cause Analysis

Two code paths exist for page generation:

1. **Legacy path** (`generator.ts` → `page-layout.ts`): Uses `buildPageLayoutPrompt()` with hardcoded rules like "Homepage should have 4-6 sections" and "Inner pages should have 2-4 sections." These rules are vague suggestions, not enforced constraints.

2. **Pipeline path** (`generate.ts` → `page-generation.ts`): Uses `buildPageGenerationPrompt()` which injects `page-design-rules.ts` constraints. The rules correctly specify section counts (e.g., Home: 5-7, Services: 4-7), but:
   - The plan phase (`plan.ts`) only includes rules as "guidelines" — the AI treats them as suggestions
   - There is no post-generation validation that the output meets the section count minimums
   - The `maxTokens: 4000` cap may truncate longer pages before all sections are generated
   - The content plan itself may only plan 2-3 sections, so even perfect generation can't fix an underplanned page

### Evidence

- User reports pages with only 2-3 components across multiple site types
- Page design rules define minimum section counts that are never validated:
  - Home: 5-7 sections (getting 2-3)
  - Services: 4-7 sections (getting 2-3)
  - About: 4-6 sections (getting 2-3)
  - Contact: 2-3 sections (appropriately thin — this is correct)

---

## Feature 1: Content Depth Enforcement

### What

Enforce minimum section counts at two stages: (a) the content plan must meet the page-type minimum, and (b) the generated output must match the planned section count.

### Acceptance Criteria

1. **Plan-level enforcement:** After the plan phase produces a `ContentPlan`, validate that each page's `sections` array meets the minimum count defined in `PAGE_DESIGN_RULES[pageType].sectionCountRange[0]`. If any page falls short, retry the plan generation with explicit feedback about which pages need more sections.
2. **Generation-level enforcement:** After generating each page, validate the output section count against the plan's section count. If sections are missing, retry with feedback identifying the gap.
3. **Token budget increase:** Increase `maxTokens` for the generate phase from 4000 to at least 6000 for pages with 5+ planned sections, to prevent truncation of longer pages.
4. **Plan prompt strengthening:** Change the plan prompt language from guidelines ("use 4-7 sections") to requirements ("you MUST produce at least N sections for this page type, this is a hard constraint").
5. **Metrics logging:** Log actual vs. expected section counts per page for every generation, so we can monitor depth compliance over time.

### Quality Bar

- Home pages: minimum 5 sections (hero + 3 middle + CTA)
- Services pages: minimum 4 sections
- About pages: minimum 4 sections
- Contact/FAQ pages: minimum 2 sections (these are utility pages, depth is not the goal)
- Generic/unclassified pages: minimum 3 sections

## Feature 2: Quality Review Agent

### What

Introduce a review agent that runs after the generate phase and before blueprint finalization. This agent evaluates each page against content depth guidelines, SEO best practices, and GEO (Generative Engine Optimization) structure — then loops back to regenerate pages that fail the quality gate.

### User Story

> As a business owner, I want my AI-generated website to have professional-quality content that ranks well in search engines and AI assistants, so that my site looks credible and attracts customers from the first day.

### Review Agent Responsibilities

The review agent evaluates each generated page against three dimensions:

#### A. Content Depth

| Check | Pass Criteria |
|-------|---------------|
| Section count | Meets page-type minimum from `PAGE_DESIGN_RULES` |
| Word count per section | Each section meets its `wordCountRange` minimum |
| Total page word count | Home >= 400 words, Services >= 350 words, About >= 350 words |
| Content specificity | No generic placeholders ("Lorem ipsum", "Your business", "Learn more") |
| Visual rhythm | No two consecutive sections use the same component type |
| Required sections present | All `required: true` sections from the page-type rule exist |

#### B. SEO Quality

| Check | Pass Criteria |
|-------|---------------|
| Meta title | 50-60 chars, includes primary keyword |
| Meta description | 150-160 chars, includes primary keyword, has CTA |
| H1 presence | Hero section contains a clear H1-equivalent heading |
| Keyword density | Target keywords appear in at least 2 sections naturally |
| Internal linking | CTA sections reference other site pages (e.g., /contact, /services) |
| Image alt text readiness | Image-capable sections have descriptive prop values (not empty) |

#### C. GEO (Generative Engine Optimization)

| Check | Pass Criteria |
|-------|---------------|
| Entity clarity | Business name, location (if provided), and industry are explicitly stated |
| Structured claims | At least 1 stat/metric/specific claim per content page (not "many years of experience" — "15 years of experience") |
| FAQ presence | At least one page includes an FAQ or accordion section (feeds AI knowledge graphs) |
| Authoritative language | Content uses first-person plural ("we") and active voice, not passive |
| Semantic completeness | Each service/offering has: what it is, who it's for, why it matters |

### Acceptance Criteria

1. **Review runs automatically** after the generate phase completes, before saving the blueprint to the database.
2. **Per-page scoring:** Each page receives a score (pass/fail per dimension + overall pass/fail). A page passes if ALL three dimensions pass.
3. **Regeneration loop:** Pages that fail are regenerated with specific feedback about what failed. Maximum 2 retry attempts per page.
4. **Feedback injection:** When a page fails review, the retry prompt includes:
   - The specific checks that failed
   - The current output as context (so the AI improves, not starts over)
   - Concrete instructions (e.g., "Add a testimonials section between the features and CTA sections")
5. **Graceful degradation:** If a page still fails after 2 retries, use the best attempt (highest score) and log the failure — don't block the entire pipeline.
6. **Review results logged:** Store review scores in the blueprint metadata so we can track quality trends.
7. **No UX impact on happy path:** The review loop should add < 15 seconds to total generation time for sites that pass on the first attempt. Failed pages add one retry cycle (~5-8 seconds each).

### Architecture Guidance

The review agent should be a **pure evaluation function** — it does not generate content itself. It:
- Takes a generated `PageLayout` + its `ContentPlanPage` + `ResearchBrief` as input
- Returns a structured score with pass/fail per check and actionable feedback strings
- Lives in `platform-app/src/lib/pipeline/phases/review.ts`
- Is called from `generate.ts` after each page generation (inside the page loop)
- Does NOT make AI calls — it is deterministic rule-based evaluation

The regeneration uses the existing `buildPageGenerationPrompt()` with appended feedback, similar to the existing prop validation retry loop.

### Non-Goals (for now)

- The review agent does not evaluate brand consistency, accessibility, or mobile layout
- The review agent does not rewrite content — it only flags issues and triggers regeneration
- No manual review UI — this is fully automated in the pipeline
- No per-section granularity in retries — entire pages are regenerated (section-level retries add complexity without proportional value)

## Impact Assessment

| Dimension | Before | After |
|-----------|--------|-------|
| Avg sections per Home page | 2-3 | 5-7 |
| Avg sections per Services page | 2-3 | 4-7 |
| SEO meta quality | Unchecked | Validated per page |
| Content specificity | Sometimes generic | Validated, no placeholders |
| Generation time | ~15-20s | ~20-30s (review adds ~5-10s) |
| Pipeline reliability | Single-pass, no safety net | Review loop with graceful degradation |
