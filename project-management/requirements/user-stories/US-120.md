# US-120: Content Hydration for Code Components

**Epic:** Code Component Quality
**Priority:** P0
**Milestone:** M29 — Content Quality

## User Story
As a site owner reviewing my generated website, I want to see real, contextual content in my preview (not generic placeholder text), so that I can evaluate whether the site accurately represents my business before provisioning.

## Acceptance Criteria
- [ ] Given a restaurant site with services "Wood-Fired Pizza", "Craft Cocktails", "Private Dining", when the features section is generated, then the feature cards display these actual service names and descriptions — not "Feature 1 Title", "Feature 1 Description"
- [ ] Given a dental clinic with testimonials in the content plan, when the testimonials section is generated, then the section shows the actual quotes, author names, and roles from the plan
- [ ] Given a site with a hero section, when the hero is generated, then the headline matches the content plan heading (e.g., "Transform Your Smile Today") not the prop description ("Main hero headline")
- [ ] Given any generated site, when the user views the preview, then zero sections contain text like "Section heading", "Description here", "Feature title", "Lorem ipsum"
- [ ] Given the content hydration phase fails, when the pipeline continues, then the site still generates successfully with current fallback content (graceful degradation)

## Dependencies
- US-111 (Live Preview Frame) — hydrated content makes the preview meaningful

## Technical Notes
- New pipeline phase: `hydrate` between `generate` and `enhance`
- Two-tier approach: direct content plan mapping first, AI generation for remaining props
- The Enhance phase (images) runs after Hydrate — image alt text benefits from hydrated content context
- Must not increase total pipeline time by more than 15 seconds
