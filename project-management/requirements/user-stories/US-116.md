# US-116: Live Regeneration with Visual Diff

**Epic:** Live Blueprint Preview
**Priority:** P1
**Milestone:** M28 — Live Blueprint Preview

## User Story
As a user who has requested a section regeneration, I want to see the new version rendered immediately in the live preview and be able to compare it with the previous version, so that I can decide whether to keep the new version or revert.

## Acceptance Criteria
- [ ] Given a section regeneration is in progress, when the API call is active, then the section in the preview shows a loading overlay with a subtle animation indicating regeneration
- [ ] Given a section regeneration completes successfully, when the new section data arrives, then the live preview renders the new version immediately without a full page reload
- [ ] Given a section has been regenerated, when the user views the edit panel, then an "Undo" button is visible that restores the previous section version
- [ ] Given a code_components section is regenerated, when the new JSX arrives, then it is transpiled and rendered in the iframe within 200ms
- [ ] Given a design_system section is regenerated, when the new component_id and props arrive, then the approximate SDC renderer updates to reflect the new component
- [ ] Given the user clicks "Undo" after a regeneration, when triggered, then the previous section's props (and JSX for code components) are restored in both the preview and the server-side blueprint

## Dependencies
- US-111 (Live Preview Frame)
- US-115 (Section Click-to-Edit — the edit panel where regeneration is triggered)
- Existing: `POST /api/blueprint/{siteId}/regenerate-section` (returns `previousSection`)

## Technical Notes
- Architecture: `architecture-live-blueprint-preview.md` Section 4.3
- The existing `regenerate-section` API already returns `previousSection` in the response — this is stored for undo
- For code_components mode, the API must also return the updated `_codeComponents.configs` entry so the preview can transpile the new JSX
- The loading overlay is rendered inside the iframe via postMessage: `{ type: "section-loading", sectionIndex, loading: boolean }`
- Undo is a client-side state operation that re-sends the old section data to both the iframe and the auto-save endpoint
- The regenerate-section route.ts needs a minor modification to include the new JSX/CSS in the response payload
