# US-044: Pipeline Data Persistence & Versioning

**Epic:** AI Content Pipeline
**Priority:** P0
**Milestone:** M7 — AI Content Pipeline
**PRD Refs:** FR-208

## User Story
As a platform developer, I want research briefs, content plans, and generation output stored as separate versioned records, so that I can re-run phases independently and debug content quality issues.

## Acceptance Criteria
- [ ] Given the Research phase completes, when the brief is stored, then it is a separate record in a `research_briefs` table linked to the site
- [ ] Given the Plan phase completes, when the plan is stored, then it is a separate record in a `content_plans` table linked to the research brief
- [ ] Given the Generate phase completes, when the blueprint is stored, then both the final blueprint and the generation metadata are preserved
- [ ] Given pipeline data persistence, when a phase is re-run, then previous versions are retained (not overwritten)
- [ ] Given versioned records, when comparing outputs, then each version has a timestamp and the model/provider used for generation
- [ ] Given pipeline data, when the Generate phase is re-run, then it can use an existing research brief and content plan without re-running those phases

## Dependencies
- US-039 (Research Phase)
- US-040 (Plan Phase)

## Technical Notes
- New Prisma schema additions: `ResearchBrief`, `ContentPlan` models with versioning
- Each record stores: content (JSONB), model used, provider, generation time, version number
- Foreign key chain: Site → ResearchBrief → ContentPlan → Blueprint
- Open Question OQ-8: Keep all versions or only latest + original? Recommend keeping all for now (storage is cheap)
- This enables the Phase Retry/Re-run feature (US-045)
