# US-083: Semantic Image Matching in Enhance Phase

**Epic:** User-Uploaded Images
**Priority:** P1
**Milestone:** M22 — User-Uploaded Images

## User Story
As a site owner, I want the AI pipeline to automatically place my uploaded images in the most relevant sections of the generated website, so that my real imagery replaces stock photos wherever it's a good fit.

## Acceptance Criteria
- [ ] Given user-uploaded images and a generated blueprint, when the enhance phase runs, then each image slot first considers user images before falling back to Pexels
- [ ] Given a user image that scores >= 0.25 match threshold against a section, when assigned, then that image is used instead of a stock photo
- [ ] Given a user image below the 0.25 threshold for all remaining slots, when no match is found, then Pexels stock photo fallback is used (existing behavior)
- [ ] Given a user image already assigned to one section, when another section is processed, then the same image is not reused (deduplication via `usedImageIds` set)
- [ ] Given each section's image assignment, when complete, then `section._meta.imageSource` is tagged as "user" or "stock"
- [ ] Given each user image match, when assigned, then `section._meta.imageMatchScore` records the confidence score
- [ ] Given no user images were uploaded (or `use_stock_only` is true), when the enhance phase runs, then behavior is identical to the current Pexels-only flow

## Dependencies
- US-082 (AI Image Description — provides tags and descriptions for matching)

## Technical Notes
- New module: `image-matcher.ts` with `rankImages()` function
- Scoring: `(tagOverlap × 0.4) + (descriptionOverlap × 0.3) + (orientationMatch × 0.15) + (subjectFit × 0.15)`
- Tag overlap: tokenize component text into keywords, intersect with image tags
- Description overlap: Jaccard similarity with stopword removal
- Orientation: 1.0 if matches, 0.3 otherwise
- Subject fit: heuristic mapping (testimonial → person/group, hero → any, product feature → product)
- `resolveImagesForSections()` gains optional `userImages` parameter — backward compatible
- Matching is pure in-memory computation (<5ms per slot), no API calls
- See `architecture-user-images.md` Sections 4.3–4.4 for full specification
