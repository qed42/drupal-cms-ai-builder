# US-117: Preview Security Hardening

**Epic:** Live Blueprint Preview
**Priority:** P0
**Milestone:** M28 — Live Blueprint Preview

## User Story
As the platform team, we want the live preview to be hardened against security threats from AI-generated code, so that malicious or erroneous JSX cannot compromise the user's session, exfiltrate data, or affect the parent application.

## Acceptance Criteria
- [ ] Given the preview iframe, when it renders, then the `sandbox` attribute is set to `allow-scripts` only (no `allow-same-origin`, no `allow-forms`, no `allow-popups`, no `allow-top-navigation`)
- [ ] Given the preview iframe srcdoc, when it is constructed, then it includes a Content-Security-Policy meta tag that allows only inline scripts, Tailwind CDN, Google Fonts, and images -- with `connect-src: 'none'` blocking all network requests
- [ ] Given AI-generated JSX containing dangerous globals (dynamic code execution, network APIs, cookie access, storage access, or parent frame access), when injected into the preview, then the sanitization layer strips these calls before transpilation
- [ ] Given a code component that attempts to call `window.parent.postMessage` with unexpected message types, when the parent receives the message, then it is rejected by a strict message type whitelist
- [ ] Given a code component with an infinite loop or excessive recursion, when rendered in the iframe, then the iframe does not freeze the parent page (separate browsing context), and can be killed by replacing the srcdoc
- [ ] Given the preview receives postMessage from a source that is not the preview iframe, when the message is processed, then it is ignored (origin validation)
- [ ] Given the security configuration, when automated tests run, then at least 5 security-specific test cases validate: XSS blocking, network isolation, DOM access prevention, prototype pollution resistance, and message validation

## Dependencies
- US-111 (Live Preview Frame -- the iframe that is being hardened)

## Technical Notes
- Architecture: `architecture-live-blueprint-preview.md` Section 5
- New file: `platform-app/src/lib/preview/__tests__/security.test.ts` -- security-focused test suite
- Sanitization function in `platform-app/src/lib/preview/transpile.ts` runs BEFORE Sucrase transpilation
- PostMessage handler in `LivePreviewFrame.tsx` validates:
  - `event.source === iframeRef.current.contentWindow`
  - `event.data.type` is in the allowed set: `["section-click", "section-hover", "ready", "error"]`
- The `referrerpolicy="no-referrer"` attribute prevents leaking the parent URL to any resources loaded in the iframe
- Defense-in-depth: the M26 Code Component Validator already blocks dangerous patterns during generation; this layer catches anything that slips through or is introduced via manual editing
- Dangerous patterns blocked by sanitization include: dynamic code execution functions, network request APIs (fetch, XMLHttpRequest), browser storage access, and parent/top window references
