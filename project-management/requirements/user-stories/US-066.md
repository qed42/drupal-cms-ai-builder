# US-066: Research Preview API Endpoint

**Epic:** Transparent AI Onboarding
**Priority:** P1
**Milestone:** M20 — AI Transparency
**Requirement:** REQ-transparent-ai-onboarding.md (Feature 2, backend)

## User Story
As the review-settings page, I need a Research preview API endpoint that runs the Research phase on demand and returns a human-readable strategy summary, so that users can see AI reasoning before generation.

## Acceptance Criteria
- [ ] Given a POST to `/api/ai/research-preview`, when valid onboarding data is provided, then the Research phase runs and returns a ResearchBrief within 20 seconds
- [ ] Given the Research preview result, when it is returned, then it includes: industry classification, target audience with pain points, tone guidance with examples, SEO keywords (8-12), and competitive positioning (without real competitor names)
- [ ] Given the result is returned, when it is stored, then it is cached on the OnboardingSession record (field: `researchPreview`) with a timestamp
- [ ] Given a cached preview exists and no inputs have changed since it was cached, when `/api/ai/research-preview` is called again, then the cached result is returned without re-running Research
- [ ] Given onboarding inputs change after a preview is cached, when the save API updates session data, then the `researchPreview` cache is invalidated (set to null)
- [ ] Given the generation pipeline starts, when the Research phase runs, then it checks for a valid cached `researchPreview` and re-uses it instead of making a duplicate AI call
- [ ] Given the API is called, when an error occurs in the Research phase, then a 200 response is returned with `{ success: false, error: "message" }` so the UI can degrade gracefully

## Dependencies
- None — extends existing Research phase logic

## Technical Notes
- The Research phase already exists at `platform-app/src/lib/pipeline/phases/research.ts`. This task wraps it in a standalone API endpoint.
- The orchestrator (`orchestrator.ts`) needs a small change: before running Research, check `session.data.researchPreview`. If valid and inputs haven't changed, skip the phase and use the cached brief.
- Input change detection: hash the relevant fields (idea, audience, industry, tone, pages, differentiators, followUpAnswers) and compare against a stored hash alongside the preview.
