# US-080: Input Quality Feedback with Archie Messaging

**Epic:** Transparent AI Onboarding
**Priority:** P2
**Milestone:** M20 — AI Transparency

## User Story
As a user filling in text fields during onboarding, I want real-time feedback that tells me how much detail Archie needs to produce great content, so that I'm motivated to provide richer inputs and understand the quality-of-input-to-quality-of-output relationship.

## Acceptance Criteria
- [ ] Given the idea step, when the user's description is short, then the feedback message reads "Give Archie more detail for a better site" instead of "Add a bit more detail for better results"
- [ ] Given the idea step, when the user's description is sufficient, then the feedback message reads "Archie has plenty to work with" instead of "Looks great!"
- [ ] Given the audience step, when the user has typed fewer than 30 characters, then a hint reads "Give Archie more detail — who exactly are your customers?"
- [ ] Given the audience step, when the user has typed 30-80 characters, then a hint reads "Good start. Adding age range, location, or pain points helps."
- [ ] Given the audience step, when the user has typed more than 80 characters, then a hint reads "Excellent. Archie can really target your messaging now."
- [ ] Given the follow-up step, when a field is empty, then a per-field hint reads "Archie needs this to write your [field] page"
- [ ] Given the follow-up step, when a field has fewer than 20 characters, then the hint reads "A bit more detail helps Archie write specific content"
- [ ] Given the follow-up step, when a field has 50+ characters, then the hint reads "Great — Archie has plenty to work with"
- [ ] Given any quality feedback message, when it renders, then it uses the existing validation/hint styling (no new components)

## Dependencies
- None — modifies existing validation messages

## Technical Notes
- Idea step: modify existing validation messages in `idea/page.tsx` — the validation logic already exists with character thresholds
- Audience step: already has character count display — add quality tier messages alongside it
- Follow-up step: currently has a single static subtitle — add per-field dynamic hints based on character length
- All messages are string changes in the respective `page.tsx` files
- No API changes needed — all quality feedback is client-side character counting
