# US-089: End-to-End Production Validation

**Epic:** Production Deployment
**Priority:** P0
**Milestone:** M23 — AWS Deployment

## User Story
As the platform team, I want to validate the full user journey on AWS — from account creation through onboarding to a live provisioned site — so that we can confirm the production environment works before any demos or external access.

## Acceptance Criteria
- [ ] Given the production URL `app.drupalcms.app`, when accessed over HTTPS, then the platform login/registration page loads within 2 seconds
- [ ] Given a new account, when the user completes the onboarding flow, then all steps work including AI analysis calls (research, plan, generate, enhance, review)
- [ ] Given a completed onboarding, when "Generate My Website" is clicked, then the blueprint generation pipeline completes successfully
- [ ] Given a generated blueprint, when provisioning triggers, then a new Drupal site is created at `{subdomain}.drupalcms.app`
- [ ] Given the provisioned site, when accessed over HTTPS, then it loads with the correct content, branding, and images
- [ ] Given the provisioning callback, when it fires, then the platform database is updated with the site status and URL
- [ ] Given Prisma migrations, when `npx prisma migrate deploy` runs on the platform container, then the database schema matches the codebase

## Dependencies
- US-086 (AWS Infrastructure Setup)
- US-087 (Wildcard DNS & SSL)
- US-088 (Production Drupal Container)

## Technical Notes
- This is a validation story, not a feature story — it exercises the existing pipeline on new infrastructure
- Environment variable differences between dev and prod are documented in `architecture-aws-deployment.md` Section 5
- Key env vars to verify: `SITE_DOMAIN_SUFFIX`, `NEXTAUTH_URL`, `DATABASE_URL`, `DB_HOST`, `JWT_SHARED_SECRET`
- Provisioning engine must correctly resolve paths on the shared Docker volume
- Drupal multisite routing verified via wildcard DNS → Caddy → Nginx → sites.php chain
