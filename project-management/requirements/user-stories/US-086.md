# US-086: AWS Infrastructure Setup

**Epic:** Production Deployment
**Priority:** P0
**Milestone:** M23 — AWS Deployment

## User Story
As the platform team, I want the AWS EC2 infrastructure provisioned with Docker Compose, so that we have a staging/production environment to deploy the platform and Drupal multisite.

## Acceptance Criteria
- [ ] Given the deployment plan, when infrastructure is set up, then an EC2 instance (t3.xlarge, 16GB RAM, 100GB EBS gp3) is running with Docker + Docker Compose installed
- [ ] Given the EC2 instance, when configured, then it has an Elastic IP assigned and security groups allow ports 80, 443, and 22 (SSH)
- [ ] Given a `docker-compose.aws.yml` file, when `docker compose up -d` is run, then 5 services start: caddy, platform, postgres, drupal-web, mariadb
- [ ] Given the platform service, when it starts, then it connects to PostgreSQL and Prisma migrations run successfully
- [ ] Given the drupal-web service, when it starts, then it runs PHP-FPM + Nginx and serves Drupal with proper file permissions
- [ ] Given the MariaDB service, when it starts, then it is accessible to the drupal-web container on port 3306
- [ ] Given a `.env.production` file, when loaded by Docker Compose, then all environment variables (DB credentials, secrets, domain suffixes) are properly injected

## Dependencies
- None

## Technical Notes
- Production Docker Compose: `docker-compose.aws.yml` with 5 services + 7 named volumes
- Drupal container: Custom `Dockerfile` based on `php:8.4-fpm-alpine` with Composer, GD, intl, opcache, pdo_mysql, zip
- Nginx config for Drupal: standard rewrite rules for clean URLs and image styles
- Platform app: `Dockerfile.prod` (existing) with production env vars
- Shared volume `drupal_site` bridges platform and Drupal for provisioning
- See `architecture-aws-deployment.md` Sections 4, 6 for full Docker Compose and Dockerfile specifications
