# US-056: Provisioning Failure Detail & Retry

**Epic:** Provisioning UX
**Priority:** P0
**Milestone:** M9 — Provisioning Hardening
**PRD Refs:** FR-404

## User Story
As a business owner, I want to see exactly which provisioning step failed and retry from that step, so that I don't have to start over when a transient error occurs.

## Acceptance Criteria
- [ ] Given a provisioning step fails, when the status API responds, then it returns the failed step name, a user-friendly error message, and an internal error detail (for debugging)
- [ ] Given a failure is displayed, when the user sees it, then the progress UI shows which step failed with an error summary
- [ ] Given a failure, when the user clicks "Retry", then provisioning re-runs from the failed step (not from the beginning)
- [ ] Given retry from a failed step, when it succeeds, then provisioning continues with subsequent steps normally

## Dependencies
- US-054 (Step-Level Provisioning Progress)

## Technical Notes
- Requires the provisioning engine to support resumption from a specific step
- Store the last completed step in the site record so retry can start from the right place
- User-friendly error messages: map internal errors to actionable messages (e.g., "Database connection timed out — this is usually temporary")
- Internal error details should be logged but only shown in an expandable "Technical Details" section
- The existing callback failure handling (recently fixed) should integrate with this
