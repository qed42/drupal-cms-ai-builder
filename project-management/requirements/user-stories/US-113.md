# US-113: Brand Token Injection

**Epic:** Live Blueprint Preview
**Priority:** P0
**Milestone:** M28 — Live Blueprint Preview

## User Story
As a user reviewing my generated website preview, I want to see my chosen brand colors and fonts applied accurately, so that the preview reflects the actual visual identity of my site.

## Acceptance Criteria
- [ ] Given a blueprint with brand tokens including colors (primary, secondary, accent, surface, text), when the preview renders, then all colors are injected as CSS custom properties (`--color-primary`, etc.) on the iframe's `:root` element
- [ ] Given a blueprint with brand fonts (heading, body), when the preview renders, then Google Fonts are loaded for both font families and applied via `--font-heading` and `--font-body` CSS custom properties
- [ ] Given a brand font that is not available on Google Fonts, when the preview renders, then the system falls back to a sensible system font stack without errors
- [ ] Given the Design Rules Engine tokens from `global.yaml` (container, typography, card, button), when the preview renders, then these tokens are applied as base styles so the overall visual rhythm matches design rules
- [ ] Given a user changes their brand colors during the style step and returns to the review page, when the preview loads, then the updated colors are reflected in the preview

## Dependencies
- US-111 (Live Preview Frame — the iframe where tokens are injected)

## Technical Notes
- Architecture: `architecture-live-blueprint-preview.md` Section 3.3
- New file: `platform-app/src/lib/preview/brand-tokens-css.ts` — converts `BrandTokens` to CSS string
- Brand tokens come from `BlueprintBundle.brand` (already populated during generation)
- Google Fonts are loaded via `<link>` tag in the iframe's `<head>` — non-blocking
- The Design Rules Engine `global.yaml` tokens define spacing, typography scales, and component patterns
- These base styles are injected as additional CSS in the iframe
- Existing code components already reference CSS custom properties (e.g., `bg-[var(--color-primary)]`) per the design rules, so no component-level changes are needed
