# US-107: Design Tokens — Cross-Section Visual Consistency

**Epic:** Design Quality & Consistency
**Priority:** P0
**Milestone:** M27 — Design Rules Engine

## User Story
As a site visitor, I want all sections on a generated site to share consistent container widths, typography scales, card styles, and button patterns, so that the site looks professionally designed rather than assembled from disconnected parts.

## Acceptance Criteria
- [x] Given global design tokens are defined, when a code component prompt is compiled, then a "Design Tokens (MUST USE)" section appears with exact Tailwind class strings
- [x] Given a healthcare site, when tokens are resolved, then softer cards (rounded-2xl) and pill buttons (rounded-full) override global tokens
- [x] Given a portfolio site, when tokens are resolved, then borderless cards, dramatic typography, and dark background alternation override global tokens
- [x] Given nested token objects (typography, button), when industry overrides only some sub-keys, then unoverridden sub-keys are preserved from global
- [x] Given no tokens are defined in any layer, when the prompt is compiled, then no tokens section appears
- [x] Unit tests cover token resolution, deep merge, and prompt compilation (9 new tests, 37 total)

## Dependencies
- US-106 (Design Rules Engine)

## Technical Notes
- Tokens cover: container, sectionSpacing, typography (h1-small + maxWidth), card, button (primary/secondary), backgroundAlternation, focus, gridGap
- Deep merge for nested objects (typography, button sub-keys) — different from array concat used for other rule categories
- Background alternation is a descriptive rule, not a class token
