# US-055: Per-Site Database Isolation

**Epic:** Provisioning UX
**Priority:** P0
**Milestone:** M9 — Provisioning Hardening
**PRD Refs:** FR-402

## User Story
As a DevOps engineer, I want each provisioned Drupal site to have its own database user with unique credentials, so that no site can access another site's data.

## Acceptance Criteria
- [ ] Given a site is provisioned, when the database step runs, then a new MariaDB user `site_{siteId}` is created with a randomly generated 32-character password
- [ ] Given the new database user, when created, then it is granted `ALL PRIVILEGES` only on the `site_{siteId}` database
- [ ] Given the generated `settings.php`, when written, then it uses the site-specific credentials (not shared root credentials)
- [ ] Given per-site database users, when applied, then this works in ALL environments including DDEV local dev
- [ ] Given the shared `root` user, when provisioning is complete, then root credentials are no longer referenced in any generated site configuration
- [ ] Given database credentials, when stored, then they exist only in the generated `settings.php` and the provisioning engine's memory during execution — never in PostgreSQL or returned by any API (NFR-10)

## Dependencies
- None (can be implemented independently)

## Technical Notes
- New provisioning step or enhancement to existing database creation step
- Password generation: use `crypto.randomBytes(32).toString('hex')` or similar
- Must handle DDEV local dev where MariaDB runs inside a container
- Monitor `max_connections` — each site uses 1-3 connections per request
- Consider: should credentials be rotatable post-provisioning? (OQ-9 — recommend deferring)
