# US-070: Store Pipeline Metadata for Transparency Features

**Epic:** Transparent AI Onboarding
**Priority:** P1
**Milestone:** M20 — AI Transparency
**Requirement:** REQ-transparent-ai-onboarding.md (Features 2-5, data layer)

## User Story
As the transparency features, I need pipeline intermediate data (ResearchBrief, ContentPlan, image search queries, input-to-section mappings) persisted alongside the blueprint, so that post-generation UI can render AI reasoning without additional API calls.

## Acceptance Criteria
- [ ] Given the Research phase completes, when its output is stored, then the full ResearchBrief (industry, audience, pain points, competitors, tone guidance, SEO keywords, compliance notes) is persisted in a `research_briefs` table or as a JSON field on the Blueprint record
- [ ] Given the Plan phase completes, when its output is stored, then the ContentPlan (page structure, section briefs, target keywords per page, global content outline) is persisted alongside the blueprint
- [ ] Given the Generate phase processes a section, when an image search query is built, then the query string is stored in the section metadata (e.g., `section._meta.imageQuery`)
- [ ] Given the Generate phase creates a section from a specific user input, when the mapping is known, then it is recorded as `section._meta.inputSource` (e.g., `{ field: "followUpAnswers.services", value: "cosmetic procedures" }`)
- [ ] Given the pipeline completes, when the Blueprint status changes to "ready", then all metadata fields are populated and accessible via the existing blueprint API
- [ ] Given the review page loads a blueprint, when it fetches blueprint data, then ResearchBrief and ContentPlan are included in the response (or available via a separate lightweight endpoint)

## Dependencies
- None — foundational data layer for all transparency features

## Technical Notes
- Currently, ResearchBrief is stored in `research_briefs` table and ContentPlan in `content_plans` table. Verify these are retained after generation completes (not cleaned up).
- Image search queries are built in `image-intent.ts` but discarded after use. Add a `query` field to the image metadata returned by the resolver.
- Input-to-section mapping is the hardest part: the Generate phase prompt creates sections from ContentPlan briefs, but doesn't explicitly tag which user input drove each section. Options:
  1. Add a `sourceInputs` field to the page generation prompt schema
  2. Post-hoc heuristic: match section headings/content against user input keywords
  3. ContentPlan already maps user inputs to sections via `contentBrief` — persist that mapping
- Recommend option 3 for v1 (least AI cost), with option 1 as a future enhancement.
