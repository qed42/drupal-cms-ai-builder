# US-088: Production Drupal Container

**Epic:** Production Deployment
**Priority:** P0
**Milestone:** M23 — AWS Deployment

## User Story
As the platform team, I want a production-ready Drupal container that replaces DDEV, so that the Drupal multisite runs in AWS with proper PHP-FPM, Nginx, and Composer-installed dependencies.

## Acceptance Criteria
- [ ] Given the Drupal container, when built from `drupal-site/Dockerfile`, then it includes PHP 8.4 FPM with extensions: GD, intl, opcache, pdo_mysql, zip
- [ ] Given the Drupal container, when running, then Nginx serves requests on port 80 with clean URL rewrite rules and image style support
- [ ] Given the multisite configuration, when a request arrives for `{subdomain}.drupalcms.app`, then Drupal's `sites.php` resolves the correct site directory and per-site `settings.php`
- [ ] Given Drupal site files, when the container runs, then `/var/www/html/web/sites` is mounted as a persistent volume so per-site files survive container restarts
- [ ] Given Composer dependencies, when the container is built, then `composer install --no-dev --optimize-autoloader` runs during image build
- [ ] Given file permissions, when the container starts, then `www-data` owns the sites directory for proper write access

## Dependencies
- US-086 (AWS Infrastructure Setup)

## Technical Notes
- Base image: `php:8.4-fpm-alpine` — minimal footprint
- Nginx config handles: clean URLs, PHP-FPM passthrough, image styles, static asset caching, dotfile blocking
- Drupal codebase and `sites.php` from the existing `drupal-site/` directory
- Volume `drupal_files` maps to `/var/www/html/web/sites` for per-site data persistence
- Volume `drupal_site` shared with platform for provisioning engine file access
- See `architecture-aws-deployment.md` Section 6 for Dockerfile and Nginx config
