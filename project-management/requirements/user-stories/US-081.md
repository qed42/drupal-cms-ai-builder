# US-081: Image Upload Step in Onboarding

**Epic:** User-Uploaded Images
**Priority:** P1
**Milestone:** M22 — User-Uploaded Images

## User Story
As a business owner going through onboarding, I want to upload my own images (product photos, team headshots, office shots) so that the generated website uses my real imagery instead of only stock photos.

## Acceptance Criteria
- [ ] Given the onboarding flow, when the user reaches the step after fonts, then a new "images" step appears with a multi-file drag-and-drop zone
- [ ] Given the upload zone, when the user drops or selects files, then it accepts .png, .jpg, .jpeg, .webp files up to 10MB each, maximum 20 files
- [ ] Given a file exceeding 10MB or an unsupported type, when upload is attempted, then a clear error message is shown and the file is rejected
- [ ] Given the upload zone, when files are dropped, then uploads begin immediately in parallel with per-file progress indicators
- [ ] Given the images step, when it loads, then a toggle "I don't have images yet — use stock photos for now" is visible at the top
- [ ] Given the stock-only toggle, when enabled, then the upload zone is hidden and the step can be skipped
- [ ] Given the images step, when it has no uploads and the toggle is off, then a "Skip" button allows proceeding without images
- [ ] Given uploaded images, when saved, then they are stored at `/public/uploads/{siteId}/images/{uuid}.{ext}` with UUID-based naming
- [ ] Given uploaded images, when the user proceeds, then `OnboardingSession.data` stores `user_images[]` and `use_stock_only` boolean

## Dependencies
- None

## Technical Notes
- Position: after `fonts` (step 9), before `follow-up` (step 10) — requires updating `onboarding-steps.ts`
- Reuses `FileUploadZone` pattern, extended for batch uploads
- Upload route extended with `uploadType: "image"` alongside existing `logo`, `palette`, `font`
- MIME type validated server-side via `file-type` magic bytes check, not just extension
- EXIF metadata stripped before storage using `sharp` library
- UUID-based filenames prevent path traversal attacks
- See `architecture-user-images.md` Section 4.1 for full interface specification
