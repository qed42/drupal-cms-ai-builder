# US-087: Wildcard DNS & Auto-SSL with Caddy

**Epic:** Production Deployment
**Priority:** P0
**Milestone:** M23 — AWS Deployment

## User Story
As the platform team, I want wildcard DNS and automatic SSL certificates for `*.drupalcms.app`, so that every provisioned Drupal site gets HTTPS automatically without manual certificate management.

## Acceptance Criteria
- [ ] Given GoDaddy DNS, when configured, then wildcard A record `*.drupalcms.app` and `app.drupalcms.app` both resolve to the EC2 Elastic IP
- [ ] Given Caddy with the GoDaddy DNS plugin, when it starts, then it automatically obtains a wildcard TLS certificate via Let's Encrypt DNS-01 challenge
- [ ] Given a request to `app.drupalcms.app`, when Caddy routes it, then it proxies to the platform Next.js app on port 3000
- [ ] Given a request to `{subdomain}.drupalcms.app`, when Caddy routes it, then it proxies to the Drupal Nginx container on port 80
- [ ] Given the wildcard certificate, when it approaches expiry, then Caddy auto-renews it without manual intervention
- [ ] Given the Caddy container, when built, then it uses a custom build with `xcaddy build --with github.com/caddy-dns/godaddy`

## Dependencies
- US-086 (AWS Infrastructure Setup)

## Technical Notes
- ADR-020: Caddy + GoDaddy DNS-01 chosen over ALB + ACM + Route 53 (saves $16/mo, no Route 53 dependency)
- Caddyfile routes: `app.drupalcms.app → platform:3000`, `*.drupalcms.app → drupal-web:80`
- Requires GoDaddy API key + secret (from https://developer.godaddy.com/keys)
- Drupal's `sites.php` continues to handle multisite resolution — no Drupal changes needed
- Custom Caddy Dockerfile needed for the GoDaddy DNS module
- See `architecture-aws-deployment.md` Sections 3–4 for Caddyfile and routing details
