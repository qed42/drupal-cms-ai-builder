# US-069: Page-Level Insights Panel in Review Editor

**Epic:** Transparent AI Onboarding
**Priority:** P2
**Milestone:** M20 — AI Transparency
**Requirement:** REQ-transparent-ai-onboarding.md (Feature 4)

## User Story
As a user reviewing a generated page, I want to see a quality summary and input-to-content mapping for the entire page, so that I understand the overall AI strategy for that page.

## Acceptance Criteria
- [ ] Given a user is viewing a page in the review editor, when they click the page-level info icon, then an insights panel opens showing: quality score (from review agent), SEO keyword coverage count, total word count, internal link count
- [ ] Given the insights panel is open, when the user views the "Your input → This page" section, then it shows a mapping of at least 3 user inputs to specific content decisions (e.g., "cosmetic procedures" → Cosmetic Dentistry section)
- [ ] Given the quality score is displayed, when the user views it, then it uses the existing 17-check review agent data stored in `blueprint.payload._review`
- [ ] Given the SEO keyword coverage, when displayed, then it shows how many of the page's target keywords appear in the content (e.g., "3/3 target keywords present")
- [ ] Given the insights panel, when it is rendered on desktop, then it appears as a slide-out panel from the page sidebar
- [ ] Given the insights panel, when it is rendered on mobile, then it appears as a full-screen overlay with a close button

## Dependencies
- Review agent data must be stored in blueprint (already implemented per TASK-290/291/292)

## Technical Notes
- Quality scores come from `blueprint.payload._review[pageSlug].review.score` and `.checks[]`.
- Input-to-content mapping needs to be built during the Generate phase: when a section is created based on a specific user input (e.g., a service mentioned in followUpAnswers), record that link in the section metadata.
- Word count and internal link count can be derived client-side from the blueprint page data.
- Target keywords per page come from ContentPlan — need to persist ContentPlan data in the blueprint or a related record.
